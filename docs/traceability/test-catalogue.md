# JobTrack test catalogue and traceability

**Closes:** Implementation plan §5.2, §5.5 exit criteria ("test categories and timeout budgets are
documented", "the requirement traceability skeleton exists").

This is a skeleton populated at M0. Every row exists before the corresponding test does; a row's
test-ID columns are filled in with the actual test method's fully qualified name as each slice is
implemented (plan §6, TDD order), never removed or renumbered retroactively — a superseded test
gets a note, not a silent renumbering, so historical traceability reports stay valid.

## 1. Test-case identifier scheme

Stable, never-reused identifiers, allocated in this document at the point a row is added (not
reused if a row is later dropped):

- **Database tests:** `TC-DB-<AREA>-<NNN>` — one PostgreSQL and one SQLite test share the same ID
  when they assert the shared contract (plan §6.6); a provider-specific test (e.g. an advisory-lock
  ordering test that has no SQLite equivalent) gets a `-PG` or `-SQ` suffix.
- **Domain/application tests:** `TC-APP-<AREA>-<NNN>`.
- **HTTP/end-to-end tests:** `TC-WEB-<AREA>-<NNN>`.
- **Golden scenarios:** `GS-<NNN>` — a golden scenario is a named, generated-or-fixed dataset used
  by multiple test IDs across categories (a golden scenario is data; a `TC-*` ID is an assertion
  against that data).

`<AREA>` is a short fixed tag, one of: `SCHEMA` `IDENT` `HIER` `LEAF` `SESSION` `PREREQ` `ACHIEVE`
`SCHED` `RATE` `COST` `AUDIT` `AUTHN` `AUTHZ` `WEB` `ROLES` `BACKUP` `TOKEN` `REQ`. Adding a new area
tag is a one-line addition to this list, not a free-form string invented per test. `REQ` covers
requester intake, holding areas, and requester progress visibility (ADR 0033).

## 2. Test categories and timeout budgets

Every automated test command runs under `gtimeout` (plan §2, CLAUDE.md). Budgets below are the
**category** ceiling used in CI (§10) and locally; an individual slow test inside a fast category is
a defect in that test, not a reason to raise the category budget.

| Category | Contains | Budget | CI job (§10) |
|---|---|---|---|
| Fast unit/property | `JobTrack.Domain.Tests`, `JobTrack.Application.Tests` (fake ports), pure interval/allocation property tests | 60s | 2 |
| Architecture/API | `JobTrack.ArchitectureTests`, `JobTrack.PublicApi.Tests`, `dotnet format --verify-no-changes` | 60s | 1 |
| Database contract (PostgreSQL) | `JobTrack.Persistence.PostgreSql.Tests`, `JobTrack.Database.ContractTests` (PostgreSQL fixtures) | 300s | 3 |
| Database contract (SQLite) | `JobTrack.Persistence.Sqlite.Tests`, `JobTrack.Database.ContractTests` (SQLite fixtures) | 180s | 4 |
| Race/concurrency (either provider) | Multi-connection race tests (§6.6): second-root, opposing moves, concurrent overlap, stale optimistic-concurrency | 300s | 3/4 |
| Identity tests | `JobTrack.Identity.Tests` | 120s | 2 |
| Web integration | `JobTrack.Web.IntegrationTests` (authZ, antiforgery, HTTP API) | 180s | 5 |
| Browser end-to-end | `JobTrack.Web.EndToEndTests` (viewport/reflow/a11y/full workflows) | 600s | 6 |
| Generated-scale/mutation/load (scheduled, not per-PR) | Property-based generated datasets at production scale, mutation coverage | 1800s | 9 |

A database-contract or race-test category timing out is presumed to be the sandbox's Unix-socket
restriction (top-level `CLAUDE.md`) before it is presumed to be a genuinely slow build — re-run with
`dangerouslyDisableSandbox: true` before treating it as a real regression.

## 5. Maintenance rule

When a schema or application slice (§6.2, §7.3) is decomposed into concrete tests, the author:

1. adds or fills in the relevant row(s) above with real test IDs (fully qualified test method
   names) before the PR implementing that slice merges;
2. does not invent a `TC-*` ID outside this table — the table is the registry; and
3. links back to this document from the test class or method's XML doc / summary comment where the
   ID is not otherwise obvious from the file's location.
