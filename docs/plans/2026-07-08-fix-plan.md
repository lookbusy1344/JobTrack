# JobTrack §8.1-§8.5 Fix Plan

**Date:** 2026-07-08  
**Scope:** Close review gaps found after implementing `docs/plans/jobtrack_impl_plan.md` through §8.5.  
**Status:** Closed — all of §2.1-§2.5 implemented and tested; verification gate (§3) passes.

## 1. Objective

Bring the ASP.NET Core web foundation, HTTP API, and browser evidence into alignment with
`jobtrack_impl_plan.md` §8.1-§8.5 without weakening the existing browser workflow coverage.

The current implementation has broad functional Razor Pages coverage and a passing web integration
suite, but it should not be treated as §8.1-§8.5 complete until the gaps below are closed.

## 2. Fix Order

### 2.1 Protect cookie-authenticated API writes from CSRF

**Finding:** The `/api/*` POST endpoints use same-origin Identity cookies and currently accept
state-changing JSON requests with only the authentication cookie. This conflicts with §8.1's cookie
auth assumption that antiforgery protection is present.

**Work:**

- Add failing direct HTTP tests proving `/api/employees/{userId}/rates/user-cost-rates`,
  `/api/employees/{userId}/rates/node-rate-overrides`,
  `/api/employees/{userId}/schedule/versions`, and
  `/api/employees/{userId}/schedule/exceptions` reject requests without a valid antiforgery token.
- Decide and implement the API antiforgery mechanism for same-origin JSON writes.
- Keep RFC 7807/problem-details behaviour for authentication, authorization, validation,
  invariant, not-found, and concurrency failures.
- Update positive API tests to send the required token explicitly.

**Acceptance checks:**

- Missing/invalid antiforgery token on every state-changing cookie-authenticated API endpoint is
  rejected.
- Authenticated requests with a valid token keep the existing success and problem-details behaviour.
- Browser form antiforgery behaviour remains unchanged.

### 2.2 Complete the §8.1 UI foundation

**Finding:** `_Layout.cshtml` still declares Bootstrap as not pinned/restored and retains default
Bootstrap scaffold markup. §8.1 requires a pinned stable Bootstrap foundation and a JobTrack design
system that does not retain an unmodified default-Bootstrap appearance.

**Work:**

- Pin the chosen Bootstrap delivery mechanism and version.
- Add JobTrack design tokens for typography, spacing, colour, focus, validation, tables,
  navigation, and status indicators.
- Replace the default scaffold layout with an operational work-map visual direction.
- Create reusable Razor partials/view components only where repetition is demonstrated.
- Keep client-side behaviour progressive and minimal.

**Acceptance checks:**

- Bootstrap is restored through a pinned mechanism and no vendored/unpinned `wwwroot/lib` tree is
  hand-authored.
- The layout no longer contains the scaffold placeholder or default visual treatment.
- Critical workflow pages use the shared design tokens/components consistently.

### 2.3 Clarify and complete the §8.4 HTTP API scope

**Finding:** The current OpenAPI surface covers only rate and schedule endpoints, while §8.5 browser
workflows cover job tree, work sessions, prerequisites, achievement, cost, audit, and administration.
If §8.4 is intended as the operational HTTP API contract, the current API is incomplete.

**Work:**

- Decide whether §8.4 for the initial release means:
  - a narrow API for schedule/rate administration only; or
  - a complete API parallel to the §8.5 operational browser workflows.
- Record the decision in an ADR or a dated plan amendment.
- If complete API coverage is required, add resource-oriented endpoints and OpenAPI tests for job
  browsing/search, structural job commands, leaf work/session commands, prerequisites, achievement,
  cost reports, audit browsing, and administrator account operations.
- Add pagination/filter/range-limit tests where list or report endpoints can grow.
- Define idempotency strategy for retry-prone commands where duplicate execution is harmful.

**Acceptance checks:**

- OpenAPI tests assert every intended endpoint and status family.
- Transport DTOs remain separate from public library contracts.
- API responses expose no database constraint names, provider details, secrets, hashes, security
  stamps, or unrestricted rate data.

### 2.4 Finish §8.2 production-security plumbing

**Finding:** Cookies, lockout, security-stamp validation, login rate limiting, HTTPS redirection,
HSTS, CSP, frame, MIME, and referrer headers exist. The plan also calls for forwarded-header trust
boundaries, secure proxy configuration, request/body limits, endpoint timeouts, cancellation
propagation, data-protection key persistence, and carefully scoped cross-origin policy.

**Work:**

- Add forwarded-header configuration with explicit trusted proxy/network configuration.
- Persist data-protection keys to a protected configurable host path outside the application
  directory.
- Add request/body size limits appropriate to the current no-attachments/plain-text content model.
- Add endpoint/request timeouts and propagate cancellation through page/API handlers where missing.
- Add a deliberate cross-origin policy baseline.
- Extend security integration tests for these behaviours where they are observable in-process.
- Document host-level settings that cannot be meaningfully proven in `WebApplicationFactory`.

**Acceptance checks:**

- Production configuration fails closed when required security paths or proxy trust settings are
  missing.
- Tests cover observable headers, size limits, timeout behaviour, and cancellation propagation.
- Operational docs identify required filesystem permissions for data-protection keys.

### 2.5 Add browser, responsive, and accessibility evidence

**Finding:** The end-to-end project currently has provider smoke coverage over HTTP, not real
browser evidence for viewport classes, keyboard operation, visible focus, zoom, orientation changes,
long content, validation messages, overflow, or accessibility checks.

**Work:**

- Select the browser automation tool and representative viewport matrix for small phone, large
  phone, tablet, and desktop.
- Add browser tests for the representative job-detail workflow first.
- Cover keyboard navigation, focus visibility, form validation, denied/missing/conflict states,
  400% zoom or equivalent text-resize evidence, and unintended horizontal overflow.
- Add automated accessibility checks as a supplement, not as the only acceptance evidence.
- Extend coverage to the remaining §8.5 workflows after the representative workflow stabilizes.

**Acceptance checks:**

- Browser tests run under `gtimeout`.
- The agreed viewport matrix passes without clipped controls or unintended page-level horizontal
  scrolling.
- Critical workflows remain operable by keyboard.
- Accessibility failures are either fixed or explicitly recorded with risk acceptance.

Run `dotnet` commands unsandboxed in this repository, as documented in `AGENTS.md`.

## 4. Non-Goals

- Do not introduce a SPA, Blazor, or bearer-token flow without a concrete non-browser API consumer.
- Do not add attachments, file uploads, or user-authored HTML.
- Do not weaken or delete existing web integration tests to make the new checks pass.
- Do not add direct SQL or domain/persistence rules to the web layer.
