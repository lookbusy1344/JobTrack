# ADR 0006: Identifier representation

**Status:** Accepted
**Closes:** Implementation plan §5.1 item 3

## Decision

Internally, every primary and foreign key is `bigint` (PostgreSQL `bigint` / SQLite `INTEGER` 64-bit), generated by the database (`GENERATED ALWAYS AS IDENTITY` on PostgreSQL; `INTEGER PRIMARY KEY` rowid aliasing on SQLite). This is the default for every entity — `JobNodeId`, `WorkSessionId`, `AppUserId`, `AuditEventId`, and so on.

Publicly, every identifier crossing the `JobTrack.Application`/`JobTrack.Abstractions` boundary is a strongly typed identifier wrapping the internal `bigint` (a `readonly record struct` per entity, e.g. `JobNodeId(long Value)`), not a bare `long`. This prevents primitive confusion (passing a `WorkSessionId` where a `JobNodeId` is expected) at compile time, consistent with the house-style value-object guidance (§5.1 item 18, §7.1).

UUIDs are not used for any identifier in the initial release. `bigint` is retained unless a specific profiling result or a genuine distribution requirement (e.g. multi-writer generation without a central sequence) later justifies a UUID for a specific entity — this is deliberately narrow: it is not a blanket future migration plan, and no code should be written today anticipating it (YAGNI).

## Consequences

- Strongly typed ID structs are defined once in `JobTrack.Abstractions` and referenced everywhere; EF Core value converters map them to/from `bigint` in both persistence providers via the shared model-configuration component (§7.4).
- HTTP route binding uses opaque strongly typed identifiers (§8.3), not raw `long` path segments — a model binder converts and validates.
- No natural keys (e.g. username) are used as primary/foreign keys anywhere; `app_user`'s natural-key uniqueness is a separate `UNIQUE` constraint, not the key itself.
- Because IDs are `bigint`, sequential ID values are not treated as a security boundary — enumeration is prevented by authorization checks (§8.3), not by ID unguessability. This is recorded because it is a common but mistaken alternative design; the threat model (§8.2) must not rely on ID opacity for authorization.
